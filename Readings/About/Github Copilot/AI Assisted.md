### Avoid Context Poisoning

Next, let's talk about context poisoning. This occurs when you get bad output in your context that the model then just can't seem to ignore. To demonstrate, I'll give a very vague prompt. Write a plan to implement shows. A show is an act playing at a venue at a particular start time. Users can add shows from the act. Now, this is a bad prompt in so many ways. It's asking the LLM to do far too much at once, and it's not giving any direction or guidance. And so it's not surprising that right off the bat it makes what I would consider an error. It's including the tenant ID as a foreign key to the show, even though the venue already has that foreign key. Now, this course is not about database design, so we won't get into that, but let's suppose that I don't catch that and I let it keep going. Now, that design mistake is part of the entity definition, and so therefore, it ends up in my database table. In fact, that mistake wove its way through all layers of the application and, more importantly, is now embedded within the context. And so let's suppose that now I've noticed the error and I scold the model and say that the show entity should not have a direct relationship with the tenant. It should inherit that relationship from the venue. A show can only be created for a venue and an act that have the same tenant. Now, this is information that I should have included in my original prompt, and in fact, I'd be better off throwing all of that work away and revising my original prompt, but let's see what happens if we allow it to continue with this poisoned context. The model has been properly chastised for its mistake, and now it's going to overcorrect. In this example, the overcorrections were sort of minor, like this comment. The show's tenant is determined by the venue's tenant. While that's true, that's not a helpful comment right here when we're talking about the venue navigation property. Other times, you'll see that context poisoning shows up as defensiveness against the original mistake. And so how do you avoid context poisoning? Well, like I said before, the most effective way is to just correct your prompt and try again, and if you notice it quickly, you can even do that within the same session, like in this case, I asked it to run the integration tests. That failed because Docker wasn't running and so I started Docker Desktop, but then I didn't just prompt the model to try again. Instead, I went back to my previous prompt and I ran it a second time. Doing so removes all of the generated output from that point new session. Just be sure to review the summary to make sure it doesn't contain the error so that you don't poison the new context. And you'll notice that sometimes the LLM chooses to summarize the chat itself, but that's not protection against context poisoning. The error might very well be in that summary, and so the best thing to do is to take control of the context yourself. Start from scratch and correct the problem in your original prompt rather than trying to scold the model.

### What Is an Agent?

Now that you have the tools for controlling context and defining project‑specific constraints, let's put those skills to use. Let's construct the building blocks of a software factory that not only accelerates but transforms your development team. That building block is an agent, or in some systems it's called a subagent or a mode. But what is an agent? Well, in one sense, it's just instructions with access to tools, but by this definition, it's little more than the project‑specific rules that we just use to define team culture. Or in another sense, an agent is the unit of work for an AI‑based solution. Each agent runs in its own isolated session, and this is how you control context. You connect agents together with inputs and outputs, forming a pipeline, and that's how you'll build your solution. With this framing in mind, let's design some agents. To design the agents that we need for our project, consider what we want to accomplish. What agents will we need to down job roles, but that's still too coarse for agents. Planning might involve writing user stories and writing specifications. You can imagine that each of these tasks has a distinct goal. User stories are focused on the business problem while specifications are focused on the technical solution, so those should be different agents. Then let's break down the tests. Writing good unit tests is ### Constrained Iteration

Now that the agents are designed to work together, let's automate the process by orchestrating workflows. You can imagine writing a deterministic function that calls one agent and then takes its output and passes it to another agent. Continue this process until the workflow is done, and in some sense, that's precisely what systems like Claude Code and Cursor do, they wrap models in deterministic algorithms, but there are advantages to also having non‑deterministic orchestrators controlling the flow. A non‑deterministic model can react dynamically to the situation in ways that are difficult to program ahead of time. Successful orchestration requires a combination of deterministic and non‑deterministic checks and controls. Some of the agents will have deterministic tools to keep them on track, while others will rely on their non‑deterministic instructions. Still, others will interact with an operator to keep the human in the loop. The process of building software, even before AI, is a process of constrained iteration. Imagine the space of all programs. This contains every program that has ever been written and anything that ever could be. Some programs are correct. They compile and they have few or no bugs, but most programs are incorrect. This space is fractal. Nearby programs differ by only a few lines of code, but some neighbors are more correct than others. Your goal is to select a correct program that does what you want. You get there by selecting a simple program and then iteratively modifying it to select more and more complex programs. When you prompt an LLM, it selects a response from the space of all possible responses. It's very likely to select one of the incorrect programs. Tools and guidance nudge it back toward a correct program, and your sequence of program increments walk it toward the one that you want. This is why vibe coding leads to poor quality solutions. Vibe coders can prompt a model incrementally closer to the program that they want, but they provide little or no constraints to keep the correctness level high. You already have the tools to do better than that, provide detailed guidance and tools to the agents. An agent takes two inputs, the user prompt and the instructions. The user prompt is for iteration. It says what you want, while the instructions are for correctness. They provide the constraints. We can take this technique to the next level by employing skills. Skills are specialized constraints that are pulled in only when the task calls for them. You can make a skill available to all agents or only to specific agents. Let me show you a scope, so you see what I mean. Skills have a description that explains when they should be used and instructions to apply in that situation. Those instructions can contain links to references, which are additional markdown documents, and they can contain scripts that perform deterministic checks. Many engines currently use skills, including Claude Code and GitHub Copilot. And in this module, I'll be showing you Kilo Code, but no matter which engine you run, they all use skills in the same way. They employ a pattern called progressive disclosure. The engine pulls in only the descriptions into context, and then it asks the LLM which of these skills are going to be most appropriate for the task at hand. Then, it loads only those skills and again asks which of these references and scripts is going to be most useful. This allows you to get very specific with your skills and have confidence that you're not going to pollute the context. Now, let me show you how I use skills in order to nudge the model up the gradient of correctness.

### Jobs, Tasks, and Workflows

We created an agent that writes user stories. Does that mean that the AI is going to take the job of the product owner? Not quite. If you look at any job, you can break it down into a set of tasks. The job of project owner takes tasks like write user story and prioritize backlog. Some tasks, like plan and increment, cannot effectively be done by AI. These are non‑compressible. Other tasks like organized tickets are easily done by AI. These are compressible. What AI does is compress the time that we spend on certain tasks, which allows us to take on more work. It also means that other people can perform those tasks with the help of AI as part of their workflows. That reduces the handoff between people where work is paused. So no, AI won't take the job of the product owner, but it will redefine it significantly. An agentic workflow is a sequence of tasks. Agents perform the compressible tasks and then pause to let a human perform the non‑compressible tasks. Orchestration is all about controlling that sequence. Let's see how to get an AI to orchestrate a workflow. An orchestrator is just another kind of agent. It's given a list of all of the other agents and a description of when to use them. It can also be given a document describing the workflow that it should run. The orchestrator looks at the task at hand coming from the user prompt and decides which agent to delegate work to. It writes task instructions to pass to the other agent in place of a user prompt. That agent runs in its own context with its own set of instructions. And when the agent is done, it reports back to the orchestrator, which then decides on the next task to delegate.
